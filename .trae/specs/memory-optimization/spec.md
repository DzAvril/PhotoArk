# 内存优化重构 Spec

## Why

当前项目在总览页面（Dashboard）和差异界面（JobDiff）存在严重的内存占用问题，每次从其他界面切换回这两个界面时内存会再次升高且不释放，导致用户体验很差。主要问题包括：

1. **全局缓存永不释放**：Dashboard使用模块级全局对象缓存数据，切换页面后数据仍驻留内存
2. **数据累积无清理**：分页加载的数据不断累积，从不清理旧数据
3. **缺少虚拟化**：长列表渲染所有元素，而非仅渲染可见区域
4. **图片/视频资源未释放**：媒体元素即使从DOM移除，浏览器仍缓存资源

## What Changes

### 高优先级变更

- **BREAKING** 移除Dashboard页面的全局缓存对象，改用React Query进行数据管理
- 实现虚拟滚动，替换当前的渲染所有项的实现
- 实现数据窗口化加载策略，限制内存中保留的数据量
- 添加组件卸载时的数据清理机制

### 中优先级变更

- 优化图片懒加载，使用Intersection Observer替代原生loading属性
- 实现图片加载队列，限制同时加载的图片数量
- 优化轮询机制，页面不可见时暂停轮询
- 使用React.memo优化频繁渲染的组件

### 低优先级变更

- 实现代码分割，延迟加载页面组件
- 添加性能监控指标

## Impact

- **Affected specs**: 前端数据管理、媒体浏览、差异对比
- **Affected code**:
  - `apps/web/src/pages/dashboard-page.tsx` - 核心重构
  - `apps/web/src/pages/job-diff-page.tsx` - 核心重构
  - `apps/web/src/pages/media-page.tsx` - 优化
  - `apps/web/src/pages/media/media-grid.tsx` - 优化
  - `apps/web/src/pages/media/use-media-browser.ts` - 优化
  - `apps/web/src/lib/api.ts` - 可能需要添加缓存层
  - 新增：`apps/web/src/hooks/use-virtual-list.ts` - 虚拟滚动Hook
  - 新增：`apps/web/src/hooks/use-data-cache.ts` - 数据缓存Hook

## ADDED Requirements

### Requirement: 数据缓存管理

系统应提供统一的数据缓存管理机制，支持自动过期和大小限制。

#### Scenario: 缓存自动过期
- **WHEN** 数据加载到缓存
- **THEN** 系统应设置过期时间（默认60秒）
- **AND** 过期后自动清理数据

#### Scenario: 缓存大小限制
- **WHEN** 缓存数据量超过阈值
- **THEN** 系统应按LRU策略清理最旧的数据
- **AND** 保留最近使用的数据

#### Scenario: 页面切换清理
- **WHEN** 用户从页面A切换到页面B
- **THEN** 页面A的非必要数据应被标记为可清理
- **AND** 在内存压力时优先清理这些数据

### Requirement: 虚拟滚动列表

系统应对长列表实现虚拟滚动，仅渲染可见区域的元素。

#### Scenario: 大数据集渲染
- **WHEN** 列表包含超过100项数据
- **THEN** 系统应仅渲染可见区域的元素（约20-30项）
- **AND** 滚动时动态加载和卸载元素

#### Scenario: 滚动性能
- **WHEN** 用户快速滚动列表
- **THEN** 滚动应保持60fps的流畅度
- **AND** 不应出现明显的卡顿或白屏

### Requirement: 数据窗口化加载

系统应实现数据窗口化加载策略，避免无限累积数据。

#### Scenario: 分页数据管理
- **WHEN** 用户加载更多数据
- **THEN** 系统应保留当前页和前后各1页的数据
- **AND** 清理超出窗口的旧数据

#### Scenario: 内存占用控制
- **WHEN** 数据加载总量超过限制（如1000项）
- **THEN** 系统应清理最早加载的数据
- **AND** 保留用户当前浏览位置附近的数据

### Requirement: 图片资源优化

系统应优化图片和视频资源的加载和释放。

#### Scenario: 图片懒加载
- **WHEN** 图片进入视口附近200px范围
- **THEN** 系统应开始加载图片
- **AND** 图片离开视口后应取消加载（如未完成）

#### Scenario: 图片加载队列
- **WHEN** 页面包含大量图片
- **THEN** 系统应限制同时加载的图片数量（最多10个）
- **AND** 按优先级队列加载图片

#### Scenario: 资源释放
- **WHEN** 图片元素从DOM移除
- **THEN** 系统应尝试释放相关资源
- **AND** 避免浏览器过度缓存

### Requirement: 轮询优化

系统应优化轮询机制，减少不必要的网络请求和内存占用。

#### Scenario: 页面可见性检测
- **WHEN** 用户切换到其他标签页或最小化窗口
- **THEN** 系统应暂停或降低轮询频率
- **AND** 页面重新可见时恢复正常轮询

#### Scenario: 轮询退避
- **WHEN** 轮询连续多次返回相同数据
- **THEN** 系统应逐渐降低轮询频率
- **AND** 数据变化时恢复正常频率

## MODIFIED Requirements

### Requirement: Dashboard页面数据加载（修改）

原实现使用全局缓存对象存储所有数据，修改后应使用React Query进行数据管理。

**修改内容**：
- 移除模块级全局对象 `dashboardCache`
- 使用React Query的缓存机制，配置staleTime和cacheTime
- 实现数据的自动重新验证和后台更新
- 组件卸载时自动清理查询缓存

### Requirement: JobDiff页面数据管理（修改）

原实现累积所有已加载的差异项，修改后应实现窗口化数据管理。

**修改内容**：
- 实现虚拟滚动，仅渲染可见项
- 限制内存中保留的数据项数量
- 提供清理按钮，允许用户手动释放数据
- 优化双面板同步滚动性能

### Requirement: Media页面媒体浏览（修改）

原实现累积所有已加载的媒体文件，修改后应优化内存使用。

**修改内容**：
- 实现图片懒加载优化
- 限制同时加载的媒体数量
- 优化Live Photo检测算法，避免重复计算
- 提供清理机制，释放已浏览的媒体资源

## REMOVED Requirements

### Requirement: 全局模块缓存

**Reason**: 全局缓存对象永不释放，导致内存泄漏
**Migration**: 使用React Query替代，提供自动清理机制

## 技术方案

### 方案一：使用React Query（推荐）

**优点**：
- 成熟的缓存管理方案
- 自动过期和重新验证
- 内置内存管理
- 开发体验好

**缺点**：
- 需要引入新依赖
- 团队需要学习新API

### 方案二：自定义缓存Hook

**优点**：
- 无需额外依赖
- 完全可控

**缺点**：
- 需要自行实现所有功能
- 可能存在边界情况bug
- 维护成本高

### 方案三：使用SWR

**优点**：
- 轻量级
- API简单

**缺点**：
- 功能比React Query少
- 某些高级场景支持不足

**推荐方案**：方案一（React Query），因为它提供了最完整的解决方案，且社区支持良好。

## 预期效果

实施这些优化后，预期达到以下效果：

1. **内存占用降低60-80%**：通过数据清理和虚拟化大幅减少内存占用
2. **页面切换流畅**：切换页面时内存能够及时释放
3. **长列表滚动流畅**：虚拟滚动确保60fps的滚动体验
4. **资源加载可控**：图片和视频加载有明确的队列和限制
5. **用户体验提升**：减少卡顿和内存警告
